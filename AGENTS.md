# Repository Guidelines

## Authority & Scope

`AGENTS.md` is the canonical repository operating guide for coding agents working in this checkout. If another agent-specific file conflicts with this one, follow `AGENTS.md` first and treat the other file as a tool-specific supplement.

Session-level user, system, and developer instructions still apply above this repository guide. For repo facts, prefer maintained source and configuration over prose when they disagree.

## Project Structure & Module Organization

This is a Swift 6 Apple-platform project generated by XcodeGen from `project.yml`. Edit `project.yml` for target, scheme, or file-structure changes, then regenerate `QwenVoice.xcodeproj`.

- `Sources/` contains the macOS Vocello app, shared models, services, SwiftUI views, and resources.
- `Sources/QwenVoiceCore/` owns shared generation contracts and runtime semantics.
- `Sources/QwenVoiceNative/`, `Sources/QwenVoiceEngineSupport/`, and `Sources/QwenVoiceEngineService/` contain the macOS native/XPC engine boundary.
- `Sources/iOS/`, `Sources/iOSSupport/`, and `Sources/iOSEngineExtension/` keep the iPhone track compile-safe.
- `QwenVoiceTests/`, `VocelloiOSTests/`, and `VocelloUITests/` hold XCTest and UI smoke coverage.
- `tests/Plans/` contains `.xctestplan` files; `docs/` contains maintained references and release notes.

## Build, Test, and Development Commands

```sh
./scripts/regenerate_project.sh
xcodebuild -project QwenVoice.xcodeproj -scheme QwenVoice build
./scripts/check_project_inputs.sh
./scripts/qa.sh validate
./scripts/qa.sh test --layer contract
./scripts/qa.sh test --layer swift
./scripts/qa.sh test --layer native
./scripts/build_foundation_targets.sh macos
./scripts/build_foundation_targets.sh ios
```

Use `./scripts/qa.sh test --layer e2e` for macOS UI smoke tests and `QWENVOICE_E2E_STRICT=1` only on a controlled Mac with Accessibility/TCC setup. The `perf` layer is opt-in and requires installed models. QA artifacts live under `build/harness/`.

## Coding Style & Naming Conventions

Match the existing Swift style: 4-space indentation, `PascalCase` types, `camelCase` properties/functions, and descriptive enum cases. Keep SwiftUI views and coordinators focused by feature area. No repo-wide SwiftFormat or SwiftLint config is present, so preserve local formatting and avoid mass-formatting, especially under `third_party_patches/`.

## Testing Guidelines

Use XCTest. Name test files after the behavior under test, for example `GenerationDraftsTests.swift`, and test methods with the `test...` prefix. Prefer the cheapest relevant lane first: contract validation for metadata, Swift tests for app logic, native tests for engine/XPC behavior, and iOS foundation builds for compile proof.

## Source Of Truth

When repo facts disagree, trust sources in this order:

1. `Sources/`
2. `project.yml`
3. `scripts/` and `.github/workflows/`
4. `docs/reference/current-state.md`, `docs/reference/engineering-status.md`, and `docs/reference/release-readiness.md`
5. other prose docs

`Sources/Resources/qwenvoice_contract.json` is the source of truth for model, speaker, variant, output, and required-file metadata. Do not reintroduce a repo-owned Python backend, Python setup path, or standalone CLI surface.

## Commit & Pull Request Guidelines

Recent commits use short imperative subjects, often scoped with a colon, such as `Engine probe Phase 2c: ...` or `Fix bench helper ...`. Keep commits focused. Pull requests should describe behavior changes, list commands run, link related issues, and include screenshots or screen recordings for UI-facing changes.
